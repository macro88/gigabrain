# Orchestration Log — Leela (Lead, Model Release Review)

## Assessment Summary

**Verdict:** Do not approve current proposal as stated. Approve narrower `dual-release-distribution` change for BGE-small only.

## Key Points

### Current Proposal Analysis
- User proposed dual-release strategy: `airgapped` variant + `online` variant + global `--model small|base|large` UX
- Surface expansion is significant: two artifact types, model family selection at init-time
- DB lifecycle impact: embedding dimensions (384 vs 768 vs 1024), vec table schema shape varies per model
- Re-embedding burden: switching models requires re-vectorizing all pages

### Approved Path (Narrower Scope)

**Branch Strategy:**
- Branch from `main`
- Proposal branch: `proposal/dual-release-distribution`
- Implementation branch: `feat/dual-release-distribution`

**Scope Limitation:**
Implement for BGE-small only. Do not add global `--model small|base|large` selection.

### Rationale

1. **Dual distribution reduces friction** — Users on systems without BGE-small precompiled binaries can fetch from GitHub Releases or use online-download slim variant.
2. **Single-model avoids DB reshaping** — Keeping BGE-small as the canonical embedding model means all init paths produce identical schema. No re-embedding tax.
3. **Future-proof** — When model-family selection is needed, it will be a DB lifecycle choice (e.g., `gbrain export` + `gbrain import --model base`), not a CLI runtime flag.
4. **Lower support burden now** — Release/install/npm paths are still single-lane. Adding model-family UX explodes test matrix (airgapped+small, online+small, airgapped+base, etc.) before release/installer processes stabilize.

## Next Steps
