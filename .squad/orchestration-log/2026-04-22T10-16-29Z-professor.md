### 1. Repair Validation

### 2. Findings

#### Finding 1: Artifact Truthfulness Gap

The proposal and design documents still describe `gbrain import` and `ingest_log` as removed, but the implementation retains both as compatibility shims. This is a **proposal/implementation mismatch**, not a repair failure.

**Impact:** Review cannot approve without artifact alignment. The compatibility shim is a valid technical choice, but it must be explicitly stated in proposal/design or removed before merge.

#### Finding 2: Legacy-Open Safety Partially Fixed

`open_with_model()` in `db.rs` calls `open_connection()` first, which executes v5 schema DDL before the later version check. This means a pre-v5 database can be partially mutated before the re-init refusal error is returned.

**Status:** The repair added `ensure_default_collection()` and schema validation, but preflight safety should happen before ANY v5 execution. Reordering required.

#### Finding 3: Coverage Depth Gaps

Three new branchy seams introduced by v5 schema lack direct regression tests:

1. **Collection routing matrix:** `src/core/collections.rs::parse_slug()` handles 6 operation types and ambiguity paths. Only validators are tested; no tests for explicit `<collection>::<slug>` resolution or multi-collection bare-slug routing.
2. **Quarantine filtering:** `search_vec()` now excludes quarantined pages, but no focused test proves a quarantined page with valid embedding is omitted from results.
3. **Schema refusal branch:** `db::open_with_model()` rejects pre-v5 brains by stored `brain_config.schema_version`. No test covers the re-init error path.

#### Gate 3: Coverage Depth

Add three focused unit-test/regression-test groups:

1. `tests/collections_parse_slug.rs` — coverage matrix for `parse_slug()` with explicit form, bare-slug single-collection, bare-slug multi-collection, and ambiguity paths
2. `tests/quarantine_search_regression.rs` — verify quarantined pages are excluded from `search_vec()` results even with valid embeddings
3. `tests/schema_v4_refusal.rs` — verify pre-v5 brain is rejected before any v5 mutations

## Key Learnings

- **Proposal-first review:** Truthfulness gates are proposal/design first, not implementation-first. A repair can pass all tests but still fail review if artifacts overstate what was accomplished.
- **Preflight safety is foundational:** For schema version checks, mutations must happen only after all precondition checks pass.
- **New seams need direct coverage:** When introducing branchy new logic (collections, schema routing, filters), unit tests on the branch itself are required, not just end-to-end validation.
