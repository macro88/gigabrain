# Orchestration Log — Bender (Tester, Model Release Review)

## Support Burden Analysis

**Verdict:** Dual-release (two variants) viable now. Global `--model` selection not viable until after migration/downgrade/installer matrix tests exist.

## Current Release State

**High-burden inputs:**
- Release/install/npm paths are single-lane (one artifact type, one init behavior)
- Installer documentation already drifts between platforms
- No downgrade/migration safety net yet
- CI/test matrix runs single-variant (embedded) only

## Pressure-Test: Two Variants (Approved)

**Scenario 1: Airgapped variant (current)**
- Platforms: Linux, macOS, Windows (all with precompiled)
- Validation: Existing CI gates + release artifact integrity check
- Cost: ~5% test overhead (duplicate packaging, signature verification)
- Risk: Low — production path already proven

**Scenario 2: Slim/Online variant**
- Init flow: First embed/query triggers HF download + cache
- Edge cases to cover:
  - Network timeout during download (postinstall hook)
  - Concurrent access during init (lock file or idempotent download)
  - Disk space exhaustion during model cache
  - HF API rate limits / outages (graceful fallback?)
- Cost: ~15% test overhead (new code path, network simulation, cleanup)
- Risk: Medium — HF reliability is external dependency

**Total burden for two variants:** Manageable. Recommend Bender Phase 3.1 rollout test plan:
1. Airgapped init (per platform)
2. Online init (simulated network, offline library mock)
3. Mixed fleet (users upgrading from v0.8 to v0.9.0, both paths in parallel)

## Why NOT Global `--model` Yet

**Missing infrastructure:**
1. **Downgrade safety** — No way to roll back from base (768) to small (384) without re-embedding
2. **Migration test matrix** — No approved test suite for init → export → import → re-embed flows
3. **Installer complexity** — npm postinstall would need conditional logic per model variant
4. **Documentation burden** — Install docs would require 6 branches (airgapped+{s,b,l}, online+{s,b,l})
5. **Rollout risk** — First-time users on large model with wrong precompile artifact = silent install failure

**Deferred infrastructure (Phase 4):**
- Round-trip export/import with schema migration
- Re-embedding coordinator (batch process, resume on failure)
- Installer matrix tests (12+ scenarios)
- Model-selection UX in install.md + quick-start.md

## Tester-Safe Rollout Sequence

**Phase 3 (current):** Airgapped + Online named channels (two artifact types, same model)
- No global `--model` flag
- User chooses at download time (artifact name)
- Init behavior differs but schema is identical

**Phase 3.1 (next):** Rollout validation plan
- CI: Both init paths tested
- Release notes: Clear guidance on when to pick slim vs. airgapped
- npm: Single package, smart postinstall
- Docs: Two parallel install sections, not model-family branches

**Phase 4 (future):** Model-family selection + re-embedding
- Only after downgrade/migration tests exist
- Only after installer matrix tests pass
- Add `--model` flag to init and export/import

## Risk Summary

- **Two variants (approved):** Risk mitigated by existing CI + new Phase 3.1 tests
- **Global model selection (deferred):** Risk too high without re-embedding + rollback infrastructure
